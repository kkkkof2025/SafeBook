# ISO 27001 实施指南

## 概述

ISO 27001 是国际公认的信息安全管理体系（ISMS）标准。通过认证不仅意味着合规，更代表组织具备了系统化的安全管理能力。本章聚焦于可落地的实施路径。

---

## 1. 实施路线图

### 1.1 六阶段实施法

```
Phase 1: 启动      Phase 2: 风险评估    Phase 3: 体系建设
   ↓                   ↓                   ↓
[领导层承诺]  →  [资产识别与评估]  →  [编写制度文档]
[组建项目组]     [风险处理计划]        [实施控制措施]
[范围界定]       [适用性声明 (SoA)]    [内部培训]

Phase 4: 运行      Phase 5: 内审       Phase 6: 认证
   ↓                   ↓                   ↓
[体系试运行]   →  [内部审核]       →  [第一阶段审核]
[持续监控]        [管理评审]           [第二阶段审核]
[纠正措施]        [改进机会]           [获取证书]
```

## 2. 风险评估方法论

### 2.1 资产识别模板

```markdown
## 信息资产登记表

| 资产ID | 资产名称 | 类型 | 所有者 | 位置 | 保密性 | 完整性 | 可用性 | 资产价值 |
|--------|----------|------|--------|------|--------|--------|--------|----------|
| A-001  | 客户数据库 | 数据 | CTO | AWS RDS | 高 | 高 | 高 | 5 |
| A-002  | CI/CD 系统 | 系统 | DevOps | GCP | 中 | 高 | 中 | 4 |
| A-003  | 员工笔记本 | 硬件 | IT | Office | 中 | 低 | 中 | 3 |
```

### 2.2 风险计算矩阵

```
风险值 = 威胁可能性 × 影响程度

影响程度 = 资产价值 × (保密性影响 + 完整性影响 + 可用性影响) / 3

威胁可能性：
  1 = 几乎不可能 (每年 <1 次)
  2 = 不太可能 (每年 1-2 次)
  3 = 可能 (每年 3-5 次)
  4 = 很可能 (每月)
  5 = 几乎确定 (每周)

风险等级：
  1-5:   低风险 (接受)
  6-10:  中风险 (监控)  
  11-17: 高风险 (处理)
  18-25: 极高风险 (立即处理)
```

## 3. Annex A 控制措施实施

### 3.1 A.8 资产管理

```yaml
资产管理:
  硬件资产:
    - 所有设备必须登记在 CMDB
    - 设备标签与责任人绑定
    - 退役设备必须安全擦除 (DoD 5220.22-M 标准)
  
  软件资产:
    - 白名单机制 (仅允许批准的软件)
    - 许可证管理 (防止合规风险)
    - 定期软件审计 (季度盘点)
  
  数据分级:
    - 公开: 可对外发布
    - 内部: 仅员工可访问
    - 机密: 需授权访问
    - 绝密: 严格的访问控制 + 审计日志
```

### 3.2 A.9 访问控制

```python
# 最小权限原则实施示例
def check_access(user, resource, action):
    """
    RBAC + ABAC 混合访问控制
    """
    # 1. 基于角色的访问控制 (RBAC)
    role_permissions = get_role_permissions(user.role)

    # 2. 基于属性的访问控制 (ABAC)
    if resource.classification == 'confidential':
        if not user.clearance_level >= resource.required_clearance:
            return False

    # 3. 环境上下文
    if action in ['delete', 'modify']:
        if not user.mfa_enabled:
            return False
        if not is_business_hours() and not user.on_call:
            return False

    # 4. 访问审计
    audit_log(user.id, resource.id, action, 'granted' if allowed else 'denied')
    return action in role_permissions
```

### 3.3 A.12 操作安全

```bash
# 恶意软件防护检查清单
maqlware_protection:
  端点:
    - EDR/XDR 部署率 >= 99%
    - 病毒库更新时间 < 24 小时
    - 扫描频率: 每日全盘 + 实时防护

  网络:
    - IDS/IPS 入侵检测启用
    - NDR (网络检测与响应)
    - DNS 安全过滤 (如 Cisco Umbrella)

  邮件:
    - SPF/DKIM/DMARC 配置
    - 附件沙箱分析
    - URL 重写与扫描

  备份:
    - 3-2-1 备份策略
    - 离线/不可变备份
    - 备份恢复测试 (每季度)
```

## 4. 文档体系构建

### 4.1 ISMS 文档金字塔

```
Level 1: ISMS 手册 (方针层)
         ├── 信息安全方针
         ├── 适用范围
         └── 角色与职责

Level 2: 策略与程序 (管理层)
         ├── 访问控制策略
         ├── 加密策略
         ├── 事件响应程序
         └── 业务连续性策略

Level 3: 操作指南 (执行层)
         ├── 账户开立/注销流程
         ├── 变更管理流程
         ├── 备份恢复操作指南
         └── 安全配置基线

Level 4: 记录与证据 (证据层)
         ├── 日志记录
         ├── 审计报告
         ├── 培训记录
         └── 风险评估表
```

## 6. 认证审核准备

### 6.1 第一阶段审核（文档审核）

审核员重点关注：
- ISMS 手册是否覆盖所有 Annex A 控制项
- 风险评估是否完整（资产列表、风险登记册）
- SoA 中对排除项是否有充分理由

### 6.2 第二阶段审核（实施审核）

常见不符合项：

1. **访问控制不严**：离职员工账户未及时注销
2. **备份测试缺失**：有备份策略但从未恢复测试
3. **培训不足**：员工安全意识培训无记录
4. **事件响应未演练**：有 IR 计划但从未桌面推演
5. **变更管理漏洞**：未经批准的配置变更
