# 共享责任模型

> 谁负责什么——云安全中最容易被误解的概念

---

## 一句话定义

**云厂商负责"云的安全"（of the Cloud），客户负责"在云中的安全"（in the Cloud）。**

这意味着：
- AWS/Azure/阿里云 负责物理安全、网络基础设施、虚拟化层
- **你**负责你的数据、IAM 配置、操作系统补丁、网络策略

---

## 按服务模型的划分

### IaaS — 客户责任最大

```
┌──────────────────────────────────────┐
│              客户负责                 │
│  ├── 应用程序                        │
│  ├── 操作系统补丁 & 配置              │
│  ├── 数据加密 & 备份                  │
│  ├── 网络 ACL / 安全组               │
│  └── IAM 权限管理                    │
├──────────────────────────────────────┤
│              云厂商负责               │
│  ├── 物理服务器                      │
│  ├── 存储基础设施                    │
│  ├── 虚拟化层                        │
│  └── 网络基础设施                    │
└──────────────────────────────────────┘
```

### PaaS — 责任各半

```
┌──────────────────────────────────────┐
│              客户负责                 │
│  ├── 应用程序代码 & 配置              │
│  ├── 数据加密 & 访问控制              │
│  └── IAM 权限管理                    │
├──────────────────────────────────────┤
│              云厂商负责               │
│  ├── 操作系统补丁                    │
│  ├── 运行时环境                      │
│  ├── 中间件                          │
│  └── 基础设施                        │
└──────────────────────────────────────┘
```

### SaaS — 客户责任最小

```
┌──────────────────────────────────────┐
│              客户负责                 │
│  ├── 数据权限                        │
│  ├── 用户账户 & MFA                  │
│  └── 数据分类 & 保留策略              │
├──────────────────────────────────────┤
│              云厂商负责               │
│  ├── 应用安全                        │
│  ├── 平台安全                        │
│  └── 基础设施安全                    │
└──────────────────────────────────────┘
```

## 常见的责任盲区

以下是实际安全事件中最常出现的"我以为云厂商会处理"的部分：

### 盲区 1：数据加密

```
❌ "我的 S3 有默认加密，数据安全的"
→ 默认加密保护的是"传输中"和"存储中"的数据
→ 但你还需要管理"谁可以解密"——这是你的责任
→ KMS 密钥管理、密钥轮换、访问审计
```

### 盲区 2：IAM 配置

```
❌ "我只给这个 Lambda 加了 S3 全权限，方便开发"
→ 结果：一个 Lambda 被利用 → 所有 S3 桶的数据被读取
→ 最小权限原则：只给这个 Lambda 需要的特定桶、特定操作
```

### 盲区 3：网络配置

```
❌ "安全组默认允许所有流量，先跑起来再说"
→ 结果：数据库端口暴露在公网 → 被自动化扫描发现 → 数据泄露
→ 真实案例：2023年某公司 Elasticsearch 暴露 → 2TB 数据泄露
```

### 盲区 4：备份与恢复

```
❌ "云厂商有冗余，不会丢数据"
→ 冗余 ≠ 备份
→ 误删操作、勒索软件加密、配置错误删除——只能靠你自己的备份策略
```

### 盲区 5：合规责任

```
❌ "上云了就自动满足合规要求了"
→ 云厂商有认证（SOC2、ISO27001）≠ 你的应用自动合规
→ 合规是共享责任：厂商提供合规的基础设施，你负责合规的配置
```

## AI 工作负载的特殊责任

当你在云上运行 AI 应用时，还有一些额外的责任：

| 责任项 | 说明 | AI 场景 |
|--------|------|---------|
| 训练数据保护 | 数据在使用中和存储中加密 | S3/OSS 中存放的训练数据 |
| 模型安全 | 模型不被窃取或篡改 | SageMaker/ModelArts 中的模型 |
| API 密钥管理 | AI API 的访问密钥 | API Key 泄露 → 被盗用推理 |
| 合规数据出口 | 训练数据是否允许出域 | 数据跨境合规 |
| 推理日志 | 推理请求的审计 | 记录谁调用了模型 |

## 最佳实践清单

- [ ] 明确你的应用使用了什么服务（IaaS/PaaS/SaaS）
- [ ] 针对每个服务，确认你和云厂商的责任边界
- [ ] 数据安全默认是你的责任——不要假设云厂商会处理
- [ ] IAM 配置遵循最小权限原则
- [ ] 定期审计谁有什么权限
- [ ] 开启云服务商的配置检查工具（AWS Config / 阿里云 Config）
- [ ] 备份策略独立于云厂商的冗余机制
