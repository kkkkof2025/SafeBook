    Web 应用架构¶

    了解一个 Web 应用是怎么组成的，你才知道漏洞会出现在哪里

    典型 Web 应用架构¶

    ┌──────────────────────────────────────────────────────┐
│                   用户浏览器                          │
│              (Chrome / Firefox / Edge)                │
└────────────────────┬─────────────────────────────────┘
                     │ HTTP/HTTPS
                     ▼
┌──────────────────────────────────────────────────────┐
│               CDN / 反向代理                          │
│          (Cloudflare / Nginx / 负载均衡)               │
└────────────────────┬─────────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────────┐
│                  Web 服务器                            │
│          (Nginx / Apache / IIS)                       │
└────────────────────┬─────────────────────────────────┘
                     │
         ┌───────────┴───────────┐
         ▼                       ▼
┌──────────────────┐   ┌──────────────────┐
│   前端应用        │   │   后端 API        │
│ (HTML/CSS/JS)    │   │ (Python/PHP/Java) │
│ (React/Vue)      │   │                  │
└──────────────────┘   └────────┬─────────┘
                                │
                    ┌───────────┴───────────┐
                    ▼                       ▼
            ┌──────────────┐       ┌──────────────┐
            │   数据库      │       │   缓存/队列   │
            │ (MySQL/PG)   │       │ (Redis/MQ)    │
            └──────────────┘       └──────────────┘

    每个层的安全风险¶

    1. 浏览器层¶

    风险 说明
    XSS 恶意脚本在浏览器执行
    CSRF 利用已登录状态发送伪造请求
    Clickjacking 透明 iframe 欺骗点击
    Cookie 窃取 通过 XSS 窃取 Session

    2. CDN / 反向代理层¶

    风险 说明
    源站 IP 暴露 CDN 配置不当，直接访问源站 IP
    SSL 配置 弱加密套件、证书问题
    缓存投毒 操纵缓存导致恶意内容被缓存

    3. Web 服务器层¶

    风险 说明
    目录遍历 访问不该访问的文件
    路径穿越 ../../../etc/passwd
    服务器信息泄露 Server 头暴露版本号
    HTTP 方法滥用 PUT/DELETE 未禁用

    4. 后端应用层（最大攻击面）¶

    漏洞 发生位置
    SQL 注入 后端处理用户输入时拼接 SQL
    RCE 后端执行系统命令时参数可控
    认证绕过 权限校验逻辑缺陷
    文件上传 上传恶意文件
    反序列化 反序列化用户数据
    SSRF 后端请求内网资源

    5. 数据库层¶

    风险 说明
    未授权访问 数据库暴露在公网
    弱密码 root/root
    SQL 注入 最经典的数据库攻击

    常见架构模式¶

    单体架构¶

    一个应用 = 一个代码仓库 = 一台/多台服务器
优点：简单
缺点：一个漏洞=全部沦陷

    微服务架构¶

    用户服务 → 订单服务 → 支付服务 → 库存服务
每个服务独立部署
优点：隔离性好
缺点：攻击面更大（服务间调用也需要安全）

    从架构看攻击面¶

    当你拿到一个 Web 应用做测试时，先搞清楚它的架构：

    1. 什么技术栈？ PHP 还是 Java？有没有框架？
    2. 登录在哪里？ 自建还是 OAuth/SSO？
    3. API 在哪里？ RESTful 还是 GraphQL？
    4. 静态文件和动态内容是否分离？
    5. 有 CDN/WAF 吗？
    6. 文件上传在哪里？怎么处理的？

    小结¶

    • Web 应用由多层组成，每一层都有自己的安全风险
    • 后端应用层是最大的攻击面
    • 做安全测试前先搞清楚目标架构，事半功倍
